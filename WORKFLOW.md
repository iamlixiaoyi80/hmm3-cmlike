## 一、工作流框架

### 8阶段开发流程

```
需求澄清 → 概念设计 → 技术方案 → 快速原型 → 核心开发 → 体验优化 → 打包发布 → 迭代规划
```

| 阶段 | 触发话术 | 产出 | 验收方式 |
|------|----------|------|----------|
| 1️⃣ 需求澄清 | "我想做个游戏..." | 需求文档 | 笑一确认：这就是我想要的 |
| 2️⃣ 概念设计 | "设计这个游戏" | GDD+原型图 | 笑一确认：看起来对 |
| 3️⃣ 技术方案 | "怎么实现？" | 技术方案书 | 用户确认：可行 |
| 4️⃣ 快速原型 | "先做个能玩的" | 可玩Demo | 笑一：好玩！继续 |
| 5️⃣ 核心开发 | "完善功能" | 完整游戏 | 用户：代码合格 |
| 6️⃣ 体验优化 | "手感不好" | 优化版本 | 笑一：舒服多了 |
| 7️⃣ 打包发布 | "怎么发给别人？" | 安装包+教程 | 笑一：朋友能玩 |
| 8️⃣ 迭代规划 | "还想加..." | 下期计划 | 笑一：就做这个 |

## 二、角色分工

### 双轨验收机制

```
游戏性验收 ←→ 代码质量验收
    ↓              ↓
  笑一负责       用户负责
```

| 角色 | 职责 | 工具 |
|------|------|------|
| 笑一（游戏制作人） | 提出想法和需求、验收游戏性、决策方向 | 自然语言描述 |
| 用户（技术负责人） | 验收代码质量、架构决策、处理技术难题 | 代码审查 |
| Claude（执行引擎） | 理解需求、生成方案、编写代码、调试测试 | 全栈开发 |

## 三、技术栈

### 推荐方案

| 类别 | 选择 | 理由 |
|------|------|------|
| 🎮 游戏引擎 | Phaser 3 | 专为2D游戏设计，API简洁，中文社区活跃 |
| 🖥️ 前端框架 | Vue 3 + Vite | 模板语法直观，Vite极速热更新 |
| 📦 状态管理 | Pinia | Vue 3官方推荐，轻量 |
| 🛠️ 构建工具 | Vite | 开发服务器启动极快，打包体积小 |
| 🚀 部署方案 | Vercel/Netlify | 零配置部署，全球CDN，免费额度充足 |

### 技术架构图

```
┌─────────────────────────────────────┐
│          Vue 3 UI Layer             │  ← 头像、背包、菜单
├─────────────────────────────────────┤
│         Phaser 3 Game Layer         │  ← 地图、战斗动画
├─────────────────────────────────────┤
│      Pinia (State Management)       │  ← 角色数据、存档
├─────────────────────────────────────┤
│         Vite Build System           │  ← 开发/打包
└─────────────────────────────────────┘
```

### 快速启动

```bash
# 1. 创建项目
npm create vite@latest h3-cmlike -- --template vue-ts

# 2. 安装依赖
cd h3-cmlike
npm install phaser pinia

# 3. 开发
npm run dev

# 4. 部署
npm run build
```

## 四、自动化测试

### 测试金字塔

```
         /\
        /  \      E2E测试 (Playwright)
       /----\     - 关键用户流程 5-10个
      /      \
     /--------\   组件测试 (Vitest)
    /          \  - 20-30个组件
   /------------\
  /              \ 单元测试 (Vitest)
 /                \ - 工具函数、状态管理
-------------------
```

### 笑一的参与方式（零代码）

笑一用中文描述测试场景，AI自动生成测试代码：

```yaml
# tests/scenarios/new-game-flow.yml
name: "新游戏流程"
author: "笑一"
priority: "critical"
steps:
  - action: "点击新游戏按钮"
    expectation: "分数归零"
  - action: "点击新游戏按钮"
    expectation: "网格重置"
```

### 两个AI协作测试模式

- **小爪**：单元测试 + E2E测试
- **Claude Code**：组件测试 + 集成测试

## 五、发布与迭代

### 版本管理（SEMVER 2.0）

```
MAJOR.MINOR.PATCH
- MAJOR: 核心玩法重构
- MINOR: 新功能
- PATCH: Bug修复
```

### 多平台发布

| 平台 | 构建方式 | 部署渠道 |
|------|----------|----------|
| Web端 | vite build | Vercel/Netlify |
| iOS端 | Capacitor | TestFlight + App Store |
| Android端 | Capacitor | Google Play |

### 热更新机制

**可热更新：** 游戏参数、资源文件、UI布局、活动配置
**不可热更新：** 核心逻辑、原生插件

### 数据分析指标

| 类别 | 指标 |
|------|------|
| 用户行为 | DAU/MAU、留存率、游戏时长 |
| 游戏平衡 | 平均得分、道具使用频率 |
| 性能 | 加载时间、FPS、崩溃率 |

## 六、潜在风险与应对

| 风险 | 应对措施 |
|------|----------|
| 上下文割裂 | 标准化中间产物格式（Schema强制） |
| 理解偏差 | 每轮开始时明确输入和输出格式 |
| 依赖管理失败 | 安装前先通知笑一确认 |
| 会话上下文丢失 | 每轮简要回顾上轮要点 |
